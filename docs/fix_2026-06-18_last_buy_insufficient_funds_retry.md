# 수정: 매일 "마지막 매수" 거부(APBK0952) — 잔고부족 시 축소 재시도

> **요약**: ① 매수 단계가 매도 직후 한도(orderable)를 1회 조회해 *합계가 한도를 넘을 때만* 비례 축소하는데, KIS `max_buy_qty`가 당일 매도대금(T+2 미결제)을 과대 반영해 합계가 한도 바로 아래여도 순서상 마지막 매수가 "주문가능금액 초과(APBK0952)"로 매일 거부됐다. ② `_execute_buy_capped`를 추가해 매수가 잔고부족으로 거부되면 한도를 실시간 재조회하고 금액을 ~3%씩 축소해 같은 종목을 재시도(최대 2회)하도록 했다 — 종목을 이연·교체하지 않고 가능한 만큼 체결한다. ③ 코드 반영 완료. 재시도 중 Slack 오류 알림은 억제하고 최종 실패만 1회 통지하며, 끝내 최소주문금액에도 못 맞추면 기존 deferred/합성 노출이 흡수한다.

- 날짜: 2026-06-18
- 파일: `trading/executor.py`
- 증상: KRW 리밸런싱에서 거의 매일 마지막 잔여 매수 1~2건이 KIS에 거부.
  - 06/15 `buy 379810 217주 @ 30,200` → `(RT_CD:7, APBK0952) 주문가능금액을 초과 했습니다`
  - 06/08·06/12·06/17 동일 패턴(둘 다/하나가 순서상 마지막 매수) → `[매수 실패] → 다음 실행 시 합성 노출로 대체`

## 원인

`rebalance()` Phase 3는 매도 완료 후 계좌별 주문가능금액을 **1회** 조회(`_fetch_krw_orderable`/`_fetch_usd_orderable`)하고, 매수 **합계 > 한도**일 때만 비례 스케일다운한다(2026-06-08·06-10 수정).

문제는 한도값 `max_buy_qty`가 당일 매도대금(T+2 결제 전 미수성 가용분)까지 끌어와 과대 계산된다는 점이다(로그의 `참고 ord_psbl_cash`는 1.3M인데 orderable은 18.7M). 그래서 합계가 한도 바로 아래(예: 06/15 98%)일 때:

1. 앞선 매수들이 실제 결제 가능 현금을 먼저 소진하고,
2. 순서상 **마지막 매수**가 남은 실현금 기준으로 재평가되어 거부된다.

사전 cap의 버퍼(×0.98)를 더 줄이는 것은 그동안 반복된 땜질이고, 과대분이 4%를 넘는 날이 있어 결정적 사전 보정이 어렵다.

## 수정

`_execute_buy_capped(ticker, currency, amount_krw, acc_name)` 추가 — Phase 3의 각 매수를 감싼다:

1. `_execute_order(..., notify_error=False)`로 1차 시도.
2. 결과가 `[오류]` + 잔고부족 키워드면: 해당 계좌 한도를 **실시간 재조회**하고
   `min(직전금액, 라이브한도) × 0.97`로 축소해 같은 종목 재시도. 최대 2회.
3. 축소액이 `min_order_krw` 미만이거나 더 줄지 않으면 중단.
4. 재시도 중 Slack 알림은 억제(`notify_error=False`), 최종 잔고부족 실패만 1회 통지.
   끝내 실패하면 **원래 계획 금액**으로 `failed_buys`에 넣어 기존 deferred/합성 노출이 흡수.

`_execute_order`에 `notify_error: bool = True` 파라미터 추가(예외 시 알림 억제용).

## 검증·주의

- 구문/임포트 OK. `_execute_buy_capped` 단위 테스트 4케이스(축소성공·계속실패·최소이하·즉시성공) 통과.
- 실주문 미실행(사용자 확인 규칙). 다음 정규 KRW 실행(10:00)에서 자연 검증.
- 재조회는 **거부 시에만** 발생(정상 경로 추가 API 호출 0) → KIS rate limit(EGW00215) 영향 최소.
- 성공 시 회전율 누적은 실제 체결(축소)액 기준, 최종 실패 이연은 원래 계획액 기준으로 분리 처리.
- 방향 안전성: 축소는 항상 한도 이하로 수렴 → 초과 거부 대신 소폭 미달(다음날 drift가 회수).
