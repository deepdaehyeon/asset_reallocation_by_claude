# 실험: 룰 하드 임계 → 소프트 라벨 A/B (2026-07-01)

## SUMMARY
- **무엇을**: detect_regime의 딱딱한 계단식 임계(예: momentum_1m>2% → 0/1 투표)를 로지스틱 램프로 부드럽게 바꿔 룰이 하드 라벨 1개 대신 확률분포를 내게 하고(`detect_regime_soft`), blend의 RULE 자리(=RF, 룰 근사기)에 주입해 하드(RF) 대비 4지표를 비교했다. 소프트 폭 scale=0(하드 앵커)·0.5·1.0·2.0 미니스윕, core30 OFF(순수 신호)+ON(라이브 현실) 양쪽, **라이브 운영점 rf_weight=0.70**(및 참고로 0.40), drift 리밸·tx·USD단일·2010~2025.
- **핵심 수치**: **불채택 — 소프트화가 4지표를 단조 악화.** 라이브 config(core30 ON·rf0.70): Martin baseline **2.17** → soft_s0.5 2.13·s1.0 2.05·s2.0 **1.98**, Ulcer 3.91 → 4.02~4.10(전부 악화), 회복 110 → 114~125일, 롤5y최악 5.0 → 4.2~5.6. 순수신호(core30 OFF·rf0.70)에선 격차 확대 — **회복 110 → 242일(2배 이상)**·Martin 1.87 → 1.70~1.80·MaxDD -11.7 → -13.3%. 부수 발견: baseline_RF가 정확한 하드 룰(soft_s0)보다도 우수(회복 110 vs 242, Martin 1.87 vs 1.75) → **RF는 단순 하드룰 재현이 아니라 유익한 학습 평활을 함**.
- **채택 여부·결론**: **불채택.** 소프트 라벨은 어느 운영점(0.40·0.70)·어느 core30 설정에서도 개선 없음이며, 룰 비중이 큰 라이브(rf0.70)에서 특히 회복기간·Martin을 뚜렷이 악화시킨다. 원인: 룰의 하드 임계는 *결정적 고확신 신호*이고, 소프트화는 진짜 전환점에서 이웃/오답 레짐으로 확률을 흘려 방어 결단력을 떨어뜨림 → 낙폭 깊어지고 회복 길어짐. 이미 연속노출(blend)+blend EWMA 평활이 whipsaw를 처리하므로 추가 소프트화는 확신만 희석. [[feedback-agility-over-turnover-reduction]](신호 둔화로 개선 안 됨)·[[feedback-regime-targets-no-tuning]] 재확인. `detect_regime_soft`·엔진 플래그(`hmm.soft_rule_scale`)는 opt-in·off로 보존(재현용).

## 배경
사용자 질문(2026-07-01): "규칙의 딱딱한 경계선(momentum 2% 등)을 소프트하게 바꾸면 어떤가?"
정정: **자산 비중은 이미 소프트**(engine.py `blend_regime_targets`가 확률가중 연속노출). 하드 라벨이 실제로 쓰이는 곳은 ① 표시용 확정 레짐, ② vol targeting·floor의 레짐 파라미터, ③ 리밸 트리거, ④ **RF 학습 타깃**(RF는 하드 룰을 베낀 지도 근사기). 따라서 "하드→소프트"의 유효 실험은 **룰의 blend 기여를 하드(RF) → analytic 소프트 룰로 교체**하는 것.

## 방법
- `detect_regime_soft(features, scale)` (regime.py): 각 하드 비교를 로지스틱 램프 `σ((x-thr)/(width·scale))`로, "count≥k" 게이트도 `σ((c-(k-0.5))/(0.4·scale))`로 소프트화. 임계·우선순위 트리는 detect_regime과 동일 → **scale→0이면 하드 룰로 수렴(nesting)**. 반환은 5레짐 확률분포(합=1).
- 엔진(backtest/engine.py): opt-in `hmm.soft_rule_scale`(None=현행 RF). 값이면 `rf_probs = detect_regime_soft(features, scale)`로 교체(RF fit 생략). blend = (1-w)·HMM + w·RULE.
- 재현: `scripts/soft_rule_ab.py`. 정합성 사전검증: 합=1, scale→0 argmax가 하드 룰과 93% 일치(불일치는 경계 충돌 케이스, 실데이터에선 더 낮음).

## 해석
1. **소프트화 = 전 지표 악화, 소프트할수록 심함**(단조). 라이브 rf0.70·core30 ON에서 Martin·Ulcer·회복·롤5y최악 모두 baseline이 최선.
2. **왜 나쁜가**: 룰의 하드 임계는 결정적 고확신 신호. 소프트화는 진짜 전환점에서 확률을 이웃/오답 레짐으로 흘려 방어 결단력을 약화 → 낙폭 깊고 회복 김(순수신호 회복 110→242일). 이미 연속노출+blend EWMA 평활이 whipsaw를 처리하므로 추가 소프트화는 확신만 희석 — 신호 둔화가 지는 기존 패턴([[feedback-agility-over-turnover-reduction]])과 일치.
3. **부수 발견 — RF ≠ 하드 룰**: baseline_RF가 정확한 하드 룰(soft_s0)보다 우수(rf0.70 OFF 회복 110 vs 242). RF(forward_window=0)는 하드 룰을 "충실히 베끼는" 게 아니라 랜덤포레스트 특유의 학습 평활로 오히려 유익한 일반화를 함 → 현행 RF 유지가 정답.
4. **core30·vol targeting 흡수**: ON에서 격차가 크게 줄어듦(회복 110→114~125) — 코어·볼타게팅이 신호 뉘앙스를 흡수하는 stagflation subregime 실험과 동일 패턴.

## 한계
- 소프트 폭(width)은 임계 스케일에 묶은 기본 1조. s0~s2.0에서 이미 단조 악화라 특정 width 최적화로 유의미한 이득을 낼 여지는 없음([[feedback-regime-targets-no-tuning]]).
- in-sample 단일기간(2010~2025). 결론이 "개선 없음/악화"라 OOS 추가검증 실익 낮음.
