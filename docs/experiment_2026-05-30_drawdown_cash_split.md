# 드로우다운 축소분 재배치 종목 분산 — 큐 항목 C 검증 (2026-05-30)

> **요약**: ① drawdown 스케일 축소분을 469830(SOL 초단기채) 100%에서 469830/305080을 50:50·70:30으로 분산하는 변형을 백테스트로 비교했다. ② 15년 중 drawdown 스케일이 단 3일(0.1%)만 발동했으며, 분산 변형의 ΔSharpe ≤ +0.001·ΔMaxDD 0.00%로 사실상 영향이 없음을 확인했다. ③ baseline(469830 전량) 유지 — 305080은 인플레 위기에서 equity와 동반 하락해 분산이 무조건 유리하지 않으며, 분산 메커니즘은 옵트인으로만 보존됐다.

## 배경

2026-05-29 트레이딩 코드 검토에서 식별한 C 항목:

> **C — 469830(SOL 초단기채)에 누적 메커니즘 중복**
> `enforce_buffer_floor`(≥7%) + 드로우다운 시 equity 축소분 이동이 모두 469830으로 수렴.
> 드로우다운+버퍼 겹칠 때 의도보다 큰 비중.

방어 자금이 초단기채 한 종목에 집중되고, deflationary 위기 시 flight-to-safety로
상승하는 305080(미국채10Y)은 전혀 활용되지 않는다는 우려.

## 코드 경로 (확인)

- 드로우다운 축소분: `trading/run.py:515` → `cash_tickers=["469830"]`로 전량 투입.
  백테스트 엔진도 동일 (`backtest/engine.py` `_apply_drawdown_scale`, 기존 `cash_ticker = "469830"` 하드코딩).
- 버퍼 플로어: `trading/run.py:526` `enforce_buffer_floor`가 469830을 ≥7%로 보정 (라이브 전용).
- 드로우다운은 **포트폴리오 기준** (`engine.py:375,520`): `dd = value/peak - 1`.

## 실험

엔진의 축소분 목적지를 config `risk.drawdown_cash_split`(티커→가중치 맵)로 파라미터화
(기본값 None = 469830 전량, 하위호환). 스크립트 `scripts/compare_drawdown_cash_split.py`.

구간: 2010-01-01 ~ 2025-04-30, W-FRI, tx_cost 0.001, FRED 매크로 10개 포함.

변형:
- `baseline`: 469830 100% (기존)
- `split_5050`: 469830 50% / 305080 50%
- `split_7030`: 469830 70% / 305080 30%

## 해석

1. **드로우다운 스케일 안전망이 15년 중 단 3일만 발동**. moderate/severe는 한 번도 안 걸림.
   레짐 기반 방어 로테이션이 포트폴리오가 -10% 떨어지기 *전에* 이미 equity를 축소하므로,
   드로우다운 스케일은 거의 작동하지 않는 최후의 보루다. 전략 자체 MaxDD가 -10.59%라
   -10% 임계를 트로프에서만 잠깐 넘는다.

2. **축소분 재배치 종목은 사실상 무의미**. baseline vs 분산안 모두 ΔSharpe ≤ +0.001,
   ΔMaxDD 0.00%. C가 우려한 "469830 집중"은 코드상 실재하나 발동 빈도가 0.1%라
   실질 영향이 무시 가능.

3. **"double accumulation" 재평가**. buffer_floor와 drawdown reduction은 합산 누적이 아니다.
   buffer_floor는 469830이 7% 미만일 때만 끌어올리므로(드로우다운이 이미 올려놨으면 no-op),
   둘이 더해져 폭증하지 않는다. 게다가 buffer_floor는 별개의 T+2 결제 버퍼 목적이고
   드로우다운 경로와 트리거가 다르다.

4. **분산이 무조건 낫지도 않다**. 305080(10Y)은 deflationary 위기(2008/2020)엔 상승하지만
   인플레 위기(2022)엔 equity와 동반 하락. 469830(초단기채)은 regime-agnostic 안전자산이라
   "전량 469830"은 보수적 설계로서 정당성이 있다.

## 결정

- **baseline(469830 전량) 유지. 라이브 config 변경 없음.** 분산이 백테스트 성과를 개선한다는
  근거가 없고, 305080 투입은 위기 유형에 대한 베팅이라 무조건 우월하지 않다.
- 분산 메커니즘은 `risk.drawdown_cash_split`로 **옵트인 보존** (기본 None = 현행 동작, 롤백 비용 0).
  미래에 포트폴리오 방어 로테이션이 실패해 큰 드로우다운이 나는 tail 시나리오용 cheap insurance로
  코드만 남겨둔다. F와 동일하게 "측정 → 문서화 → baseline 유지" 패턴.

## 남은 메모

  무해하나, 라이브에서도 분산을 쓰려면 이 함수를 다중 배분으로 고쳐야 한다. 라이브 리스크 코드
  변경은 사용자 승인 후 별도 작업으로 둔다.
